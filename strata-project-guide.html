1. The Big Picture

What is this project, and what is it trying to do?

What is Strata?

Strata is a Minecraft mod — a program that modifies and extends the game. Specifically, it's built for the Fabric mod loader (one of the main tools people use to run mods on Minecraft Java Edition). The mod is written in Java, which is the same language Minecraft itself is built in.

What does it do?

Strata is designed as a modular ecosystem — a family of related mods that share a common foundation and update together. Right now it has two modules: strata-core (shared tools and systems that other modules use) and strata-world (custom biomes and an in-game biome editor). The vision includes future modules for structures, RPG systems, and an in-game asset creation tool.

Why is it split into modules?

Think of it like LEGO sets. Each set (module) can work on its own, but they all share the same base plate (strata-core). If you only want custom biomes, you only need strata-core + strata-world. This modular approach keeps things organized and means you can develop, test, and update each piece independently.

2. Key Concepts for Beginners

Terms and ideas you'll see throughout this project.

Analogy: A Restaurant Kitchen

Think of this project like a restaurant. Java is the language the recipes are written in. Gradle is the kitchen equipment that combines raw ingredients (your code) into a finished dish (a runnable mod). Fabric is the restaurant's management system — it knows how to take your finished dish and serve it to the customer (Minecraft). A Mixin is like sneaking into the restaurant's original recipe book and penciling in changes to an existing recipe without rewriting the whole thing.

Java

The programming language used to write the mod. Java code lives in .java files. Each file typically defines a class — think of a class as a blueprint for a thing (like a blueprint for a "Biome Editor" or a "Magic Wand"). Minecraft itself is written in Java, which is why mods for Java Edition are also written in Java.

Gradle (the Build System)

Gradle is a tool that takes all your code and builds it — meaning it compiles the Java into a format the computer can run, packages it up, and downloads any libraries you need. Files like build.gradle, settings.gradle, and gradle.properties are all instructions for Gradle. You don't edit these often — they mostly configure "how to build this project."

Fabric (the Mod Loader)

Fabric is the system that loads your mod into Minecraft. It's like a plugin manager for the game. Fabric provides APIs (pre-built tools and functions) that make it easier to add things to Minecraft without rewriting the whole game. The file fabric.mod.json tells Fabric: "Here's my mod, here's what it's called, and here are the entry points where my code starts running."

Mixins

Mixins are a powerful (and somewhat advanced) technique. They let you inject your own code into Minecraft's existing code at specific points, without having direct access to Minecraft's source files. For example, Strata uses a mixin to change what color the water appears when you're in the biome editor. It's like intercepting a phone call and adding your own message before passing it along.

Packages & Namespaces

Java files are organized into packages — like folders within folders. This project uses io.strata.core and io.strata.world as its base packages. The deep folder nesting you see (like src/main/java/io/strata/core/) is just Java's way of organizing code by its package name. Each dot in the package name maps to a folder.

JSON (JavaScript Object Notation)

JSON files store structured data in a human-readable format. In this project, they define things like biome properties (what colors, what temperature), mod metadata, language translations, and world presets. They're the "data" that your Java code reads and acts on.

4. Root-Level Files

The files sitting at the very top of the project.

README.md

The Project's Front Page

WHAT A Markdown document that introduces the project. It describes the philosophy ("most mod ecosystems fragment over time — Strata solves this"), lists all planned modules with their status, and links to the architecture docs.

WHY Every open-source or collaborative project has a README. It's the first thing anyone sees when they open the project. It answers: "What is this? What does it do? How do I get started?"

.gitignore

Git's "Ignore List"

WHAT A list of file patterns that Git (your version control system) should not track. Includes .gradle/, build/, IDE files, OS junk files like .DS_Store, and Minecraft run directories.

WHY You don't want to upload thousands of auto-generated build files or personal IDE settings to your repository. That would be like uploading your kitchen trash along with your recipe book. This file keeps the repo clean.

5. Gradle & the Build System

Files that control how your code gets compiled and packaged.

Analogy: Gradle = The Factory

Your Java code is the raw material. Gradle is the factory that processes it. build.gradle is the factory blueprint. gradle.properties is the spec sheet ("use this version of Minecraft, this version of Java"). settings.gradle is the list of product lines ("we make strata-core and strata-world"). The gradlew script is the "start the factory" button.

settings.gradle

Module Registry

WHAT Tells Gradle: "This project is called 'strata' and it contains two sub-modules: strata-core and strata-world." Also configures where Gradle downloads Fabric's build tools from.

WHY Without this file, Gradle wouldn't know the sub-folders are separate buildable modules. It's the master manifest for the whole multi-module project (called a "monorepo" — everything in one repository).

build.gradle (root)

Shared Build Configuration

WHAT Defines settings that apply to all modules: which version of Minecraft to target, which Fabric API version to use, how to compile Java, and how to package the final mod JARs. It uses the fabric-loom plugin, which is Fabric's special build tool.

WHY Instead of duplicating the same Minecraft version, Java version, and Fabric settings in every module, you write it once here and all modules inherit it. DRY principle — "Don't Repeat Yourself."

HOW The subprojects { ... } block runs for each module. It applies the Fabric Loom plugin, sets the Minecraft/Yarn/Fabric dependencies, and configures Java compilation to target Java 21.

gradle.properties

Version Numbers & Settings

WHAT A simple key-value file holding all version numbers: Minecraft (1.21.11), Fabric loader (0.18.4), Yarn mappings, Cloth Config, Strata itself (0.1.0), and Java version (21). Also sets Gradle's memory limit to 2GB.

WHY When a new Minecraft version drops, you update version numbers in one file instead of hunting through multiple build scripts. Think of it as the project's central "settings panel."

gradlew & gradlew.bat

Gradle Launcher Scripts

WHAT Shell scripts that download and run the correct version of Gradle automatically. gradlew is for Mac/Linux, gradlew.bat is for Windows. The gradle/wrapper/ folder contains the tiny JAR that makes this work.

WHY This guarantees everyone building the project uses the exact same Gradle version, regardless of what's installed on their computer. It eliminates "works on my machine" problems.

strata-core/build.gradle

Core Module's Build Config

WHAT Additional dependencies specific to strata-core: Cloth Config (a library for creating in-game settings screens) and JUnit (a testing framework). It also bundles Cloth Config inside the mod JAR so players don't have to install it separately.

WHY Each module can have its own unique libraries. Core needs Cloth Config for settings; other modules might need different things. Keeping this separate from the root build.gradle keeps things organized.

strata-world/build.gradle

World Module's Build Config

WHAT Declares that strata-world depends on strata-core (project(':strata-core')), plus its own copy of Cloth Config and JUnit for testing.

WHY This line — modImplementation project(':strata-core') — is the dependency chain in action. It means strata-world can use any class from strata-core. It's like saying "this department needs access to the shared services department."

6. strata-core — The Shared Foundation

Everything that other modules rely on. Think of it as the "shared library" or "common toolkit."

Analogy: The Shared Toolbox

If each Strata module is a builder working on a different part of a house, strata-core is the shared toolbox. It has the logging system (a notebook everyone writes in), the event system (a walkie-talkie network), the config system (shared blueprints), and the wand registry (a universal remote that routes to the right tool). No module reinvents these — they all reach into the same toolbox.

Entry Points (Where the mod starts)
strata-core/src/main/java/io/strata/core/StrataCore.java

Server-Side Entry Point

WHAT The very first code that runs when Minecraft loads this mod on the server side (the part that manages game logic). It initializes the config system, the attachment system (for storing data on players), the event system, and the wand handler registry. It also detects when a player joins for the very first time.

WHY Every Fabric mod needs an entry point — a class that implements ModInitializer. This is Fabric's way of knowing "run this code when the game starts." Think of it as the mod's main() function.

HOW Fabric reads the fabric.mod.json file, finds the entry point class listed there, and calls its onInitialize() method. That method then calls all the setup functions in a specific order.

strata-core/src/main/java/io/strata/core/StrataCoreClient.java

Client-Side Entry Point

WHAT The client-side counterpart. "Client" means the part of the game that runs on your computer and handles rendering, graphics, and user interface. Currently this file is mostly empty — a placeholder for future client-only setup.

WHY Minecraft has two "sides": server (game logic, world data) and client (graphics, input, UI). Some code should only run on one side. Having separate entry points lets you keep that clean.

strata-core/src/main/java/io/strata/core/StrataVersion.java

Version Constants

WHAT A simple file that stores the mod's version number ("0.1.0"), the mod's unique ID ("strata_core"), and a helper method to look up the current Minecraft version from Fabric.

WHY Having version info in one centralized place means other parts of the code can display it (like in startup banners or debug screens) without hardcoding the version everywhere.

Configuration System
strata-core/src/main/java/io/strata/core/config/StrataCoreConfig.java

Core Settings Definition

WHAT Defines the settings that strata-core offers: verboseLogging (should the mod print extra debug info?) and showStartupBanner (show the fancy startup message?). Both are simple on/off toggles.

WHY Giving users configurable settings makes the mod flexible. Someone debugging a problem can turn on verbose logging; someone who finds the startup banner annoying can turn it off. These get saved to a TOML file on disk so they persist between game sessions.

strata-core/src/main/java/io/strata/core/config/StrataConfigHelper.java

Configuration Utility

WHAT A helper that wraps the Cloth Config / AutoConfig library. Provides simple register() and get() methods so any module can load and save configuration files without dealing with the library's complexity directly.

WHY Instead of every module writing boilerplate code to set up configuration, they call one method from this helper. It's a "wrapper" — a thin layer that makes a complex thing simpler to use.

Event System
strata-core/src/main/java/io/strata/core/event/StrataEvents.java

Central Event Bus

WHAT Defines all the custom events that Strata modules can "fire" (trigger) and "listen to" (react to). Events include: Strata fully initialized, player joined for the first time, player respawned, player data loaded/saving, and a custom asset was registered.

WHY Events are a way for different parts of the code to communicate without knowing about each other. Module A says "hey, a player just joined for the first time!" and Module B can react to that without Module A ever importing Module B's code. This keeps modules loosely coupled (independent).

HOW Uses Fabric's EventFactory to create typed event hooks. Any module can register a "listener" (a function that runs when the event fires). When the event fires, all registered listeners are called in order.

strata-core/src/main/java/io/strata/core/event/callback/*.java

Event Callback Interfaces (6 files)

WHAT Each file defines a single interface (a "contract" or "shape") for one event type. For example, PlayerFirstJoin.java says: "any listener for this event must accept a ServerPlayerEntity parameter." The six callbacks are: StrataInitialized, PlayerFirstJoin, PlayerRespawn, PlayerDataLoaded, PlayerDataSaving, and AssetRegistered.

WHY Interfaces define the "shape" of the conversation. They're like saying: "If you want to listen for the 'player first join' event, you must provide a function that takes a player as input." This makes the code type-safe — the compiler catches mistakes before the game even runs.

Registries & Wand System
strata-core/src/main/java/io/strata/core/registry/StrataRegistry.java

Block/Item/Entity Registration Helper

WHAT Provides simple methods to register new blocks, items, and entity types with Minecraft's internal registry. Methods like registerBlock() and registerItem() take a mod ID and a name, and handle all the boilerplate of creating properly namespaced identifiers.

WHY Minecraft requires everything (blocks, items, entities) to be "registered" with a unique ID like strata_world:strata_wand. This helper hides the repetitive registration code so modules can register things in one line.

strata-core/src/main/java/io/strata/core/wand/StrataWandRegistry.java

Wand Handler Registry

WHAT A central registry where each module registers its "wand handler" — the code that should run when a player right-clicks with the Strata Wand. When the wand is used, this registry checks all registered handlers to find which one(s) match the current context (what the player is looking at, where they are, etc.).

WHY The Strata Wand is a universal tool — one wand, many editors. This registry is what makes that possible. strata-world registers a "biome editor" handler; a future strata-structures module could register a "structure editor" handler. The wand asks the registry "who wants to handle this?" and routes accordingly.

strata-core/src/main/java/io/strata/core/wand/WandHandler.java

Wand Handler Interface

WHAT An interface (contract) that every wand handler must implement. It requires four methods: getId() (unique name), getDisplayName() (human-readable name), matches() (should this handler activate?), and handle() (do the thing).

WHY By defining this interface in strata-core, any module can create a wand handler without knowing about other modules. It's a "plugin system" for the wand.

Utilities
strata-core/src/main/java/io/strata/core/util/StrataLogger.java

Logging Utility

WHAT A unified logging system with info(), warn(), error(), and debug() methods. Debug messages only print when verboseLogging is enabled in the config. All log messages are prefixed with "Strata" so you can find them easily in log files.

WHY Logging is essential for debugging. Instead of each file creating its own logger, they all use this shared one with consistent formatting and respect for the verboseLogging setting.

strata-core/src/main/java/io/strata/core/util/StrataNbtHelper.java

NBT Data Utility

WHAT Helper methods for working with NBT (Named Binary Tag) data — Minecraft's format for saving structured data to disk. Provides convenience methods for saving/loading identifiers and string lists.

WHY NBT is Minecraft's native data format (used for player data, chunk data, item data, etc.). Writing raw NBT code is tedious and error-prone, so this helper simplifies common operations.

strata-core/src/main/java/io/strata/core/attachment/StrataAttachments.java

Player Data Attachments

WHAT Defines persistent data that gets attached to each player using Fabric's attachment system: PLAYER_INITIALIZED (has this player been set up before?) and PLAYER_TAGS (a flexible key-value map for future use).

WHY Minecraft doesn't natively support custom per-player data. Fabric's attachment system lets you "attach" data to players that automatically saves and loads with the world. This is how the mod remembers "has this player seen the welcome message?"

7. strata-world — Biomes & World Generation

Custom biomes, world generation, and the in-game Biome Editor.

Analogy: The World-Building Department

If strata-core is the shared toolbox, strata-world is the landscaping team. It creates new types of terrain (biomes), provides an in-game editor so players can design biomes visually, and hooks into Minecraft's world generation engine to place custom biomes in the game world. The "mixins" are like the team reaching into Minecraft's terrain-painting machinery and adding their own colors to the palette.

Module Entry Points
strata-world/src/main/java/io/strata/world/StrataWorld.java

Server-Side Entry Point

WHAT The main initialization file for strata-world. Sets up everything in a specific order: registers configuration, initializes biomes, hooks into world generation, registers the Strata Wand item, registers the biome editor as a wand handler, sets up networking, and configures first-join behavior (auto-gives the wand and shows a welcome message in the special "Biome Design World").

WHY Just like strata-core has its entry point, each module needs its own. This is where strata-world "plugs into" the game. The careful ordering matters — you can't register a wand handler before the wand registry exists, for example.

strata-world/src/main/java/io/strata/world/StrataWorldClient.java

Client-Side Entry Point

WHAT Handles client-only concerns: listening for the server's "open the editor" message and displaying the editor screen, saving editor drafts when the player disconnects, restoring drafts when they reconnect, and ticking the preview zone manager every frame.

WHY The biome editor UI is entirely client-side — it's your screen, your mouse clicks, your color sliders. The server just says "open the editor" and the client handles everything visual. This separation is important in multiplayer (the server doesn't render graphics).

Configuration & Biome Registry
strata-world/src/main/java/io/strata/world/config/WorldConfig.java

World Module Settings

WHAT Configuration options for strata-world: master on/off toggle, biome frequency (how common custom biomes are), noise parameters for the "Verdant Highlands" biome, and editor settings like debounce delays and undo stack depth.

WHY Makes the mod customizable without changing code. Server admins can tweak biome rarity; users can adjust how many undo steps the editor keeps. "Debounce" means "wait a moment before applying changes" — this prevents the game from constantly regenerating chunks while you're dragging a slider.

strata-world/src/main/java/io/strata/world/biome/StrataBiomes.java

Biome Registry Keys

WHAT Defines the identity of each custom biome as a "registry key" — currently just one: VERDANT_HIGHLANDS (a rolling, forested hill biome). The actual biome data (colors, temperature, etc.) lives in the JSON files; this Java file just declares "a biome with this name exists."

WHY Minecraft needs to know about biomes at startup so it can reference them during world generation. This file creates the "name tag" for each biome; the JSON file fills in the details.

World Generation
strata-world/src/main/java/io/strata/world/worldgen/StrataWorldgen.java

Biome Injection into World Gen

WHAT The core world generation logic. Contains the math that determines where custom biomes appear in the world using Minecraft's "multi-noise" system (biomes are placed based on temperature, humidity, continentalness, etc.). Reads values from the config file.

WHY This is what actually makes custom biomes appear in newly generated terrain. Without this, you'd have a biome definition that the game never places anywhere.

HOW Minecraft generates terrain using "noise parameters" — mathematical values that vary across the world map. Each biome occupies a region of this parameter space. This code adds Strata's biomes to that parameter list, so the game knows "if temperature is X and humidity is Y, place Verdant Highlands here."

strata-world/src/main/java/io/strata/world/worldgen/StrataWorldEvents.java

World Gen Event Hooks

WHAT The "orchestrator" for world generation events. Gets called by the mixin (see below) and delegates to StrataWorldgen. Also registers any biome modifications via Fabric's BiomeModifications API.

WHY Keeps the mixin code minimal (mixins should be as small as possible since they're injected into Minecraft's code) and delegates to clean, testable Java methods.

strata-world/src/main/java/io/strata/world/worldgen/StrataWorldFeatures.java

Biome Feature Modifications

WHAT Reserved for modifying features (trees, flowers, ores) in vanilla or custom biomes. Currently empty — Strata biomes define their features directly in their JSON files to avoid a known Fabric bug with "feature order cycles."

WHY Having this file ready means the architecture is prepared for when feature modifications are needed. The comment in the code explains the technical reason for the current approach.

Mixins (Modifying Minecraft's Code)
strata-world/src/main/java/io/strata/world/mixin/VanillaBiomeParametersMixin.java

Biome Injection Mixin

WHAT Injects code at the end of Minecraft's method that writes overworld biome parameters. After vanilla finishes adding all its biomes, this mixin appends Strata's biomes to the list.

WHY Fabric's API doesn't provide a native way to add overworld biomes (as of MC 1.21). This mixin is necessary to "sneak" custom biomes into vanilla's biome list. It's the least-invasive approach — it doesn't modify any existing biome, just adds to the end of the list.

strata-world/src/main/java/io/strata/world/mixin/BiomeMixin.java

Visual Override Mixin (Editor)

WHAT Intercepts Minecraft's biome color methods (getWaterColor(), getGrassColorAt(), getFoliageColor(), hasPrecipitation()) and returns the editor's values instead when the biome editor is open.

WHY This is how the editor provides instant visual preview. When you change the water color slider in the editor, this mixin makes all water in the world immediately reflect that new color — no world regeneration needed. It's "Layer 1" — fast visual changes.

strata-world/src/main/java/io/strata/world/mixin/EnvironmentAttributeMapMixin.java

Sky/Fog Override Mixin (Editor)

WHAT Similar to BiomeMixin but targets sky color, fog color, and water fog color. In Minecraft 1.21, these colors moved to a different system (EnvironmentAttributeMap), so they need a separate mixin.

WHY Minecraft's internals change between versions. Sky/fog colors used to be on the Biome object directly, but were moved. This mixin targets the new location so the editor's color overrides still work.

The Biome Editor
strata-world/src/main/java/io/strata/world/editor/BiomeEditorScreen.java

The Editor UI

WHAT The main editor window — a full-screen interface with a header bar (biome name, ID, action buttons), a sidebar with five tabs (Visual, Terrain, Features, Spawns, Export), and a content area. Supports keyboard shortcuts (Ctrl+Z for undo, Ctrl+Y for redo, arrow keys and Tab for navigation).

WHY This is the heart of the mod's user experience. Instead of manually writing JSON files to create biomes (which is what you'd normally have to do), players get a visual, interactive editor.

HOW Extends Fabric's Screen class, which provides the basic framework for custom game screens. The render() method draws everything each frame. The mouseClicked() and keyPressed() methods handle input.

strata-world/src/main/java/io/strata/world/editor/BiomeEditorState.java

Editor State (The Data Model)

WHAT Holds all the data for the biome being edited: colors (sky, fog, water, grass, foliage), weather (rain/snow), terrain noise parameters, feature list, mob spawn entries, display name, biome ID, export status, and an undo/redo manager. Can serialize itself to/from JSON for draft saving.

WHY Separating the data (state) from the UI (screen) is a fundamental design pattern. The state can be saved, loaded, copied, undone, and serialized independently of how it's displayed. The screen just reads from the state and writes to it.

strata-world/src/main/java/io/strata/world/editor/BiomeEditorSession.java

Editor Session Manager

WHAT A singleton (only one instance exists at a time) that tracks whether the editor is currently open, and holds references to the active editor state and preview zone manager. The mixins check BiomeEditorSession.isActive() to decide whether to apply color overrides.

WHY The editor state needs to be accessible from multiple places: the screen, the mixins, the preview system, the network handler. This singleton acts as the central "is the editor open, and if so, what's the current state?" accessor.

strata-world/src/main/java/io/strata/world/editor/tabs/*.java

Editor Tab Panels (6 files)

WHAT Each tab is a separate class handling one section of the editor:

EditorTab.java — Abstract base class that all tabs extend.
VisualTab.java — Color picker with RGB sliders for sky, fog, water, grass, foliage colors + weather toggles (rain/snow).
TerrainTab.java — Sliders for the six multi-noise parameters (temperature, humidity, etc.) that control terrain shape.
FeaturesTab.java — Add/remove terrain features (trees, flowers, ores) by identifier.
SpawnsTab.java — Add/remove/edit mob spawn entries (entity type, weight, group size).
ExportTab.java — Export the biome as JSON, save as a datapack, copy to clipboard, or save a draft.

WHY Splitting the editor into tabs keeps each piece manageable. Each tab only deals with its own domain. It also makes the UI navigable — you don't get overwhelmed by seeing every setting at once.

strata-world/src/main/java/io/strata/world/editor/BiomeDesignWorldPreset.java

Special World Type Detection

WHAT Registers and detects the "Strata: Biome Designer" world preset — a special world type in the "Create World" screen specifically designed for testing biomes. It can detect whether the current world was created using this preset.

WHY The mod behaves differently in a "Biome Design World" — it auto-gives the wand, shows a welcome message, and is singleplayer-only. This class handles detecting that special context.

strata-world/src/main/java/io/strata/world/editor/StrataWand.java

The Wand Item

WHAT The physical in-game item that opens editors. When right-clicked, it ray-casts (shoots an invisible line) 5 blocks from the player, queries the wand registry for matching handlers, and invokes the appropriate one. If multiple handlers match, it picks the first (future versions will show a disambiguation menu).

WHY The wand is the user-facing entry point to the entire editor ecosystem. One item, one right-click, and you're in the appropriate editor for your context. It's designed to scale — as more modules add handlers, the wand automatically routes to them.

strata-world/src/main/java/io/strata/world/editor/BiomeEditorWandHandler.java

Biome Editor Wand Handler

WHAT Implements the WandHandler interface from strata-core. ID: "biome_editor". Currently always matches (any wand use opens the biome editor). Sends a network packet to the client to open the editor screen.

WHY This is the "glue" between strata-core's wand system and strata-world's biome editor. It tells the wand registry: "I exist, my name is 'biome_editor', and when activated, tell the client to open the biome editor."

strata-world/src/main/java/io/strata/world/editor/PreviewZoneManager.java

Live Preview Coordinator

WHAT Manages the debounced (delayed) regeneration of chunks when the editor's terrain settings change. Distinguishes between "Layer 1" changes (visual, fast) and "Layer 2" changes (structural, slow). Has configurable delay timers and captures undo snapshots before regenerating.

WHY Regenerating terrain chunks is expensive. You don't want to regenerate every time a slider moves by one pixel. The debounce system waits until you stop moving the slider, then regenerates once. Layer 1 (colors) can update instantly via mixins; Layer 2 (terrain shape) needs actual chunk regeneration.

strata-world/src/main/java/io/strata/world/network/OpenBiomeEditorPayload.java

Network Packet

WHAT A server-to-client network message that carries no data — it's just a signal saying "open the biome editor." Uses Fabric's payload system with a unique ID (strata_world:open_biome_editor).

WHY In Minecraft, the server decides what happens (game logic), and the client decides what to show (UI). The wand handler runs on the server, but the editor screen opens on the client. This packet bridges that gap. It's like the server sending a text message to the client saying "show the editor now."

8. Documentation Files

Design documents, specs, reviews, and workflow guides.

docs/ARCHITECTURE.md

11. Resource Files (JSON & Metadata)

Data files that Minecraft and Fabric read to know what this mod provides.

strata-core/src/main/resources/fabric.mod.json

Core Module Identity Card

WHAT Tells Fabric: "This mod is called strata_core, version 0.1.0. Its server entry point is StrataCore and its client entry point is StrataCoreClient. It requires Fabric Loader 0.18+, Fabric API, Minecraft ~1.21, and Java 21+."

WHY This is the mod's "passport." Without it, Fabric wouldn't know the mod exists or how to load it. Every Fabric mod must have this file.

strata-world/src/main/resources/fabric.mod.json

World Module Identity Card

WHAT Same concept as above but for strata-world. Additionally declares a dependency on strata_core and references the mixin configuration file.

strata-world/src/main/resources/strata_world.mixins.json

Mixin Configuration

WHAT Tells the mixin system which mixin classes to load and which Minecraft classes they target. Lists BiomeMixin, EnvironmentAttributeMapMixin, and VanillaBiomeParametersMixin.

WHY Mixins need to be declared in a configuration file so the mixin processor knows to apply them. It's like a manifest saying "inject these modifications into these parts of Minecraft."

strata-world/src/main/resources/assets/strata_world/lang/en_us.json

English Language Strings

WHAT All user-facing text in English: the world type name ("Strata: Biome Designer"), item names ("Strata Wand"), tooltips, welcome messages, tab labels, and screen titles.

WHY Minecraft uses "language files" for all displayed text. This makes the mod translatable — someone could create a fr_fr.json for French without changing any Java code. It also keeps user-facing text out of the code, making it easier to update wording.

strata-world/src/main/resources/data/strata_world/worldgen/biome/verdant_highlands.json

Verdant Highlands Biome Definition

WHAT The complete data definition for the Verdant Highlands biome: temperature, downfall, colors (sky, fog, water, grass, foliage), sound effects, and terrain features (what trees, plants, and ores generate there).

WHY Minecraft's world generation is "data-driven" — biomes are defined in JSON, not code. This means you can tweak biome properties without recompiling the mod. The Java code registers the biome key; this JSON fills in all the details.

strata-world/src/main/resources/data/strata_world/worldgen/world_preset/biome_designer.json

Biome Designer World Preset

WHAT Defines the "Strata: Biome Designer" world type that appears in the Create World screen. Configures a special overworld setup designed for biome testing.

strata-world/src/main/resources/data/minecraft/tags/worldgen/world_preset/normal.json

World Preset Tag

WHAT Adds the Biome Designer world preset to Minecraft's list of "normal" world types so it appears in the world creation UI alongside Default, Superflat, etc.

WHY Minecraft uses "tags" to group things. The "normal" tag controls which world types appear in the Create World screen. Without adding our preset to this tag, players wouldn't be able to select it.

12. How It All Connects

The big picture: how data and control flow through the project.

Startup Flow

When Minecraft launches with the mod installed, here's what happens in order:

1. Fabric reads both fabric.mod.json files and discovers two mods
2. Fabric calls StrataCore.onInitialize() → sets up config, events, wand registry
3. Fabric calls StrataWorld.onInitialize() → registers biomes, worldgen, wand, editor handler
4. On the client side, StrataCoreClient and StrataWorldClient set up client-only features
5. When generating terrain, VanillaBiomeParametersMixin fires → Strata biomes get added
6. When a player joins for the first time in a Biome Design World → they get the wand + welcome message

Editor Flow

When a player right-clicks the Strata Wand:

1. StrataWand.use() fires on the server
2. Queries StrataWandRegistry for matching handlers
3. BiomeEditorWandHandler matches → sends OpenBiomeEditorPayload to client
4. Client receives packet → creates BiomeEditorState (restoring draft if one exists)
5. Opens BiomeEditorScreen with tabs
6. BiomeEditorSession becomes active → mixins start applying color overrides
7. Player edits colors/terrain → Layer 1 changes apply instantly, Layer 2 changes debounce
8. Player exports → JSON file written to world's datapack folder
9. Player closes editor or disconnects → draft saved to disk

The Layer 1 / Layer 2 System

Layer 1 (Visual, instant) — Sky color, fog, water color, grass, foliage, weather. These changes are applied immediately through mixins that intercept Minecraft's rendering. No terrain regeneration needed. Controlled by BiomeMixin and EnvironmentAttributeMapMixin.

Layer 2 (Structural, debounced) — Temperature, humidity, noise parameters, features, mob spawns. These require actual chunk regeneration to take effect. The PreviewZoneManager waits for the player to stop adjusting sliders before regenerating, to avoid wasting performance.

The Module Dependency Chain

strata-core depends on nothing (except Fabric + Minecraft)
strata-world depends on strata-core
Future modules (strata-structures, strata-rpg) will also depend on strata-core
strata-creator (future) will depend on everything, as it's the universal in-game design tool

This is a classic "layered architecture" — each layer can only depend on the layers below it, never above. This prevents circular dependencies and keeps the codebase clean.
